## Что это за проект

`AML Task Data` - это SQL-проект для PostgreSQL, который описывает базу данных системы управления задачами. По смыслу это учебная или прототипная модель таск-трекера уровня Jira/YouTrack/Trello, но с акцентом не только на задачи, а на полноценную серверную модель данных:

- пользователи и их настройки;
- проекты и участники проектов;
- роли участников внутри проектов;
- приглашения пользователей в проекты;
- типы задач: эпик, история, задача, ошибка, подзадача;
- статусы и workflow внутри проекта;
- спринты;
- задачи, подзадачи, связи между задачами;
- наблюдатели задач;
- история смены статусов;
- комментарии;
- вложения;
- списание времени;
- уведомления;
- бизнес-журнал активности;
- технический аудит изменений;
- роли PostgreSQL, права доступа и Row Level Security.

То есть проект показывает не просто набор таблиц, а подход к проектированию базы для приложения, где важны целостность данных, права доступа, история изменений и аналитические запросы.

## Главная идея

Главная идея базы - хранить рабочие процессы команды вокруг задач так, чтобы приложение могло отвечать на практические вопросы:

- какие задачи есть в проекте;
- кто над ними работает;
- в каком они статусе;
- какие задачи просрочены;
- сколько времени было потрачено;
- кто и когда менял данные;
- какие действия происходили в проекте;
- какие данные может видеть конкретный пользователь.

Модель построена вокруг проекта. Проект является верхним бизнес-контейнером: у проекта есть владелец, участники, статусы, спринты, задачи и активность. Пользователь может участвовать в разных проектах с разными ролями.

Задача является центральной рабочей сущностью. У задачи есть тип, статус, исполнитель, автор, приоритет, сроки, оценки, родительская задача, спринт и история изменений. Это позволяет использовать базу и для Kanban-подхода, и для Scrum-подхода.

## Какие задачи может решать модель
Эта база подходит для демонстрации следующих сценариев:

- создание и настройка проекта;
- добавление участников в проект;
- разграничение ролей: владелец, администратор, участник, наблюдатель;
- создание задач разных типов;
- ведение спринтов;
- отслеживание текущего статуса задач;
- хранение истории смены статусов;
- учет трудозатрат;
- хранение комментариев и вложений;
- уведомления пользователей;
- аудит технических изменений;
- проверка доступа к данным на уровне строк PostgreSQL.

Важный момент: часть логики намеренно находится в базе, а не только в приложении. Например, база сама проверяет допустимые значения ролей, статусов, приоритетов, форматы ключей проектов, положительность оценок и корректность временных диапазонов. Это снижает риск того, что некорректные данные попадут в систему через баг в приложении или прямой SQL-доступ.

## Технологии и требования

Проект рассчитан на PostgreSQL.

Используемые расширения:

- `pgcrypto` - нужен для `gen_random_uuid()`, чтобы генерировать UUID внутри базы;
- `citext` - нужен для email без чувствительности к регистру.

Почему `citext` важен: email `User@example.com` и `user@example.com` с точки зрения пользователя обычно один и тот же адрес. `citext` позволяет сделать уникальность email корректнее, чем обычный `text` или `varchar`.

Проверялось по локальной DataGrip-настройке с PostgreSQL 16.x, но сами скрипты используют стандартные возможности PostgreSQL и должны работать на современных версиях PostgreSQL, где доступны `pgcrypto`, `citext`, RLS и `security_invoker` у views.

## Структура файлов

Проект разбит на несколько SQL-файлов. Их порядок важен.

| Файл | Назначение |
| --- | --- |
| `01_extensions_and_schema.sql` | Создает расширения, схему `aml_task`, таблицы, базовые ограничения и функцию сборки `full_name`. |
| `02_indexes.sql` | Создает индексы для основных сценариев чтения, фильтрации и аналитики. |
| `03_functions_and_triggers.sql` | Создает функции, триггеры `updated_at`, `row_version`, технический аудит и audit-триггеры. |
| `04_seed_base.sql` | Заполняет базу демонстрационными данными: пользователи, проект, статусы, спринт, задачи, комментарии, уведомления, логи времени. |
| `05_demo_queries.sql` | Содержит готовые аналитические и проверочные SELECT-запросы. |
| `06_security.sql` | Создает PostgreSQL-роли, права, представления, helper-функции и RLS-политики. |
| `06_security.local.sql` | Локальный файл с dev-login ролями и паролями. Не коммитится. |
| `.gitignore` | Запрещает коммитить локальные настройки IDE и файлы с секретами. |

## Порядок запуска

Рекомендуемый порядок запуска:

```sql
01_extensions_and_schema.sql
02_indexes.sql
03_functions_and_triggers.sql
04_seed_base.sql
06_security.sql
06_security.local.sql
05_demo_queries.sql
```

Почему именно так:

1. Сначала нужны расширения, схема и таблицы.
2. Потом создаются индексы по уже существующим таблицам.
3. Потом создаются функции и триггеры, потому что они ссылаются на таблицы.
4. Потом добавляются демо-данные.
5. Потом включаются права, представления и RLS.
6. Потом локально создаются dev-login роли с паролями.
7. После этого можно запускать демо-запросы.

`05_demo_queries.sql` не меняет схему и нужен для проверки данных и аналитики.

## Локальные пароли

Реальные локальные пароли не лежат в коммитируемом `06_security.sql`.

Они вынесены в файл:

```text
06_security.local.sql
```

Этот файл подходит под правило из `.gitignore`:

```gitignore
*.local.sql
```

Значит, при обычном `git add .` файл с локальными паролями не попадет в Git.

Сейчас локальный файл создает dev-роли:

| Роль | Пароль | Назначение |
| --- | --- | --- |
| `aml_admin_dev` | `admin123` | Локальный администратор. |
| `aml_app_dev` | `app123` | Локальная роль приложения. |
| `aml_reader_dev` | `reader123` | Локальная read-only роль. |

Это именно dev-пароли для локальной разработки. Для публичного репозитория или продакшена такие значения использовать нельзя.

## Общая архитектура данных

Логически базу можно разделить на несколько блоков.

### Пользователи и доступ

Таблицы:

- `users`;
- `user_sessions`;
- `project_members`;
- `project_invitations`;
- `notifications`.

Этот блок отвечает за идентичность пользователя, сессии, членство в проектах, приглашения и персональные уведомления.

### Проектная модель

Таблицы:

- `projects`;
- `statuses`;
- `sprints`;
- `issue_types`.

Этот блок описывает проект как рабочее пространство. У каждого проекта свой набор статусов и спринтов. Типы задач глобальные, потому что типы вроде "задача", "ошибка", "эпик" обычно одинаковы для всей системы.

### Задачи и работа

Таблицы:

- `issues`;
- `issue_watchers`;
- `issue_status_history`;
- `issue_relations`;
- `comments`;
- `attachments`;
- `time_logs`.

Этот блок хранит саму работу: задачи, связи, обсуждение, вложения и время.

### Аудит и логи

Таблицы:

- `activity_log`;
- `audit_log`.

`activity_log` хранит бизнес-события: задача создана, статус изменен, проект создан. Это журнал, который можно показывать пользователю.

`audit_log` хранит технические изменения строк: INSERT, UPDATE, DELETE, старое и новое состояние записи. Это журнал для администрирования, расследований и контроля данных.

## Основные сущности

### `users`

Пользователи системы.

Ключевые поля:

- `id` - UUID пользователя;
- `email` - уникальный email в типе `citext`;
- `password_hash` - хэш пароля, не открытый пароль;
- `first_name`, `last_name`, `full_name` - имя пользователя;
- `settings` - JSONB-настройки пользователя;
- `is_active` - активность учетной записи;
- `last_login_at` - последнее посещение;
- `row_version` - версия строки для optimistic locking;
- `created_at`, `updated_at`, `deleted_at` - технические даты.

Почему есть `deleted_at`: используется soft delete. Пользователь не удаляется физически сразу, чтобы не ломать историю задач, комментарии, аудит и связи.

Почему есть `settings jsonb`: пользовательские настройки часто меняются по структуре. JSONB позволяет добавлять новые параметры без миграции таблицы.

### `user_sessions`

Сессии пользователей и refresh token hash.

Важное решение: хранится не refresh token, а `refresh_token_hash`. Это снижает риск при утечке базы. Даже если таблицу прочитают, сам токен не должен быть восстановим напрямую.

Поля `is_revoked`, `revoked_at`, `expires_at`, `last_used_at` позволяют:

- завершать отдельную сессию;
- видеть активные устройства;
- автоматически считать сессию истекшей;
- анализировать последнее использование.

### `projects`

Проекты являются главным контейнером задач.

Ключевые поля:

- `name` - название проекта;
- `project_key` - короткий ключ проекта, например `AMLTASK`;
- `owner_id` - владелец проекта;
- `is_archived` - архивный проект;
- `deleted_at` - soft delete.

`project_key` ограничен форматом:

```text
^[A-Z][A-Z0-9_]{1,19}$
```

Это позволяет строить человекочитаемые коды задач вида:

```text
AMLTASK-1
AMLTASK-2
AMLTASK-3
```

### `project_members`

Участники проекта и их роли.

Допустимые роли:

- `owner`;
- `admin`;
- `member`;
- `viewer`.

Почему членство вынесено в отдельную таблицу: один пользователь может быть в нескольких проектах, а его роль может отличаться в каждом проекте.

`left_at` и `is_active` позволяют не удалять историю членства физически. Это полезно для аудита и аналитики.

### `project_invitations`

Приглашения пользователей в проект.

Допустимые статусы:

- `pending`;
- `accepted`;
- `declined`;
- `expired`;
- `revoked`.

Зачем отдельная таблица приглашений: приглашенный email может еще не быть пользователем системы. Нельзя привязать приглашение только к `user_id`, потому что пользователь может зарегистрироваться позже.

### `issue_types`

Справочник типов задач.

Демо-типы:

- `task`;
- `bug`;
- `story`;
- `epic`;
- `subtask`.

Поле `is_subtask_allowed` задает бизнес-правило: не каждый тип задачи должен позволять подзадачи. Например, для `bug` и `subtask` в seed-данных подзадачи запрещены.

### `statuses`

Статусы задач внутри проекта.

У статуса есть:

- `name`;
- `category`;
- `position`;
- `color`;
- `is_default`;
- `is_final`.

Категории:

- `todo`;
- `in_progress`;
- `done`.

Почему есть и `name`, и `category`: название статуса может быть кастомным, например `Code Review`, `QA`, `Ready for Deploy`, но для аналитики важно понимать общую категорию: работа еще не начата, идет или завершена.

`position` задает порядок колонок на доске.

### `sprints`

Спринты проекта.

Допустимые статусы:

- `planned`;
- `active`;
- `completed`;
- `cancelled`.

Спринт содержит `start_date` и `end_date`, а ограничения проверяют корректность дат.

### `issues`

Центральная таблица задач.

Ключевые поля:

- `project_id` - проект;
- `issue_number` - номер задачи внутри проекта;
- `type_id` - тип задачи;
- `status_id` - текущий статус;
- `sprint_id` - спринт;
- `parent_issue_id` - родительская задача для подзадач;
- `title`, `description` - содержание задачи;
- `reporter_id` - автор;
- `assignee_id` - исполнитель;
- `priority` - приоритет;
- `story_points` - оценка сложности;
- `due_date` - срок;
- `rank_position` - позиция в списке/на доске;
- `original_estimate_minutes` - первоначальная оценка времени;
- `remaining_estimate_minutes` - оставшаяся оценка;
- `row_version` - версия строки;
- `deleted_at` - soft delete.

Уникальность `(project_id, issue_number)` позволяет иметь независимую нумерацию задач в каждом проекте.

Допустимые приоритеты:

- `lowest`;
- `low`;
- `medium`;
- `high`;
- `highest`;
- `critical`.

### `issue_watchers`

Наблюдатели задач.

Таблица связывает пользователя и задачу. Составной primary key `(issue_id, user_id)` запрещает добавить одного и того же наблюдателя к одной задаче дважды.

### `issue_status_history`

История смены статусов.

Эта таблица нужна, потому что текущий статус в `issues.status_id` отвечает только на вопрос "где задача сейчас". Для аналитики нужны ответы:

- когда задача попала в статус;
- сколько времени она провела в статусе;
- кто перевел задачу;
- какой был предыдущий статус;
- какой комментарий был к переходу.

Поля `entered_at` и `left_at` позволяют считать lead time и cycle time.

### `issue_relations`

Связи между задачами.

Типы связей:

- `blocks`;
- `is_blocked_by`;
- `relates_to`;
- `duplicates`;
- `is_duplicated_by`;
- `depends_on`;
- `is_dependency_for`;
- `clones`;
- `is_cloned_by`.

Ограничение запрещает связать задачу саму с собой.

### `comments`

Комментарии к задачам.

Комментарии имеют `deleted_at`, то есть тоже поддерживают soft delete. Это сохраняет историю обсуждения и позволяет скрывать комментарий без физического удаления строки.

### `attachments`

Вложения к задачам.

Хранится не сам файл, а ссылка:

- `file_name`;
- `file_url`;
- `mime_type`;
- `file_size_bytes`;
- `checksum_sha256`.

Такое решение обычно правильнее для приложений: файлы лежат в объектном хранилище или файловом сервисе, а база хранит метаданные.

### `time_logs`

Списание времени по задачам.

Хранит:

- задачу;
- пользователя;
- дату работы;
- количество минут;
- описание.

Время хранится в минутах, а не в дробных часах. Это упрощает суммирование и исключает ошибки округления.

### `notifications`

Уведомления пользователей.

`payload jsonb` позволяет хранить разные типы уведомлений без создания отдельной таблицы на каждый тип.

Состояние прочтения задается через `is_read` и `read_at`.

### `activity_log`

Бизнес-журнал событий.

Примеры событий:

- `project_created`;
- `issue_created`;
- `status_changed`.

Это журнал для интерфейса и бизнес-аналитики. Его можно показывать в ленте активности проекта.

### `audit_log`

Технический аудит изменений.

Хранит:

- имя таблицы;
- операцию: `INSERT`, `UPDATE`, `DELETE`;
- primary key записи в JSONB;
- старые данные;
- новые данные;
- пользователя, который сделал изменение;
- время изменения;
- transaction id.

Из аудита специально убираются чувствительные поля:

- `users.password_hash`;
- `user_sessions.refresh_token_hash`.

Это важное решение: аудит нужен для контроля, но он не должен становиться местом утечки секретов.

## Бизнес-решения

### project - главная граница

Большинство бизнес-данных привязано к `project_id`. Это позволяет проверять доступ по членству в проекте.

Пользователь видит проектные задачи, статусы, спринты и активность только тогда, когда он участник проекта или владелец.

### Нумерация задач внутри проекта

Задачи не имеют глобального человекочитаемого номера. Вместо этого используется пара:

```text
project_key + issue_number
```

Пример:

```text
AMLTASK-1
```

Такой формат удобнее для пользователей, потому что по коду сразу видно, к какому проекту относится задача.

### Soft Delete вместо физического удаления

Для важных сущностей используется `deleted_at`.

Это дает несколько преимуществ:

- история не ломается;
- аудит остается связанным с данными;
- можно восстановить запись;
- аналитика не теряет контекст.

Физическое удаление используется только там, где это безопасно или естественно, например для каскадных зависимостей внутри проекта.

### Row Version для конкурентных обновлений

В таблицах `users`, `projects`, `statuses`, `sprints`, `issues`, `comments` есть `row_version`.

Она увеличивается триггером при обновлении. Это нужно для optimistic locking: приложение может проверить, что пользователь редактирует актуальную версию строки, а не перезаписывает чужие изменения.

### `updated_at` Заполняется базой

`updated_at` обновляется триггером. Это надежнее, чем заставлять приложение каждый раз помнить про это поле.

### Статус как справочник

Текущий статус задачи хранится ссылкой на `statuses`, а не строкой.

Это дает:

- единый список статусов на проект;
- контроль порядка колонок;
- категории для аналитики;
- возможность переименовать статус без изменения всех задач.

### История статусов отдельно от текущего статуса

`issues.status_id` - это текущее состояние.

`issue_status_history` - это временная история переходов.

Такое разделение делает текущие запросы быстрыми, но сохраняет возможность строить аналитику по жизненному циклу задачи.

### Трудозатраты в минутах

Трудозатраты хранятся как `integer` в минутах. Это проще, точнее и надежнее, чем дробные часы.

Часы можно получить в SELECT:

```sql
ROUND(SUM(time_spent_minutes) / 60.0, 2)
```

### JSONB там, где может меняться структура

JSONB используется в:

- `users.settings`;
- `notifications.payload`;
- `activity_log.details`;
- `audit_log.old_data`;
- `audit_log.new_data`;
- `audit_log.record_pk`.

Это разумно для полуструктурированных данных, где набор полей может отличаться от записи к записи.

### Ограничения

В модели много `CHECK`, `UNIQUE` и `FOREIGN KEY`. Это сделано осознанно: база должна быть последней линией защиты данных.

Примеры:

- нельзя создать задачу с пустым названием;
- нельзя создать проект с неправильным ключом;
- нельзя создать отрицательную оценку;
- нельзя указать `left_at` раньше `joined_at`;
- нельзя создать связь задачи самой с собой;
- нельзя создать две задачи с одинаковым номером в одном проекте.

## Безопасность и доступы

Файл `06_security.sql` делает несколько вещей:

- создает NOLOGIN-роли;
- отзывает доступы у `PUBLIC`;
- выдает минимально нужные права;
- создает безопасные представления;
- включает Row Level Security;
- создает RLS-политики.

### Роли

Основные роли:

| Роль | Тип | Назначение |
| --- | --- | --- |
| `aml_owner` | NOLOGIN | Владелец схемы/объектов. |
| `aml_admin` | NOLOGIN | Административный доступ. |
| `aml_app` | NOLOGIN | Доступ приложения. |
| `aml_readonly` | NOLOGIN | Ограниченное чтение. |

Локальные LOGIN-роли создаются в `06_security.local.sql`:

| LOGIN-роль | Наследуемая роль |
| --- | --- |
| `aml_admin_dev` | `aml_admin` |
| `aml_app_dev` | `aml_app` |
| `aml_reader_dev` | `aml_readonly` |

### Почему NOLOGIN и LOGIN разделены

NOLOGIN-роли описывают набор прав. LOGIN-роли используются для подключения.

Так проще управлять доступами:

- права задаются один раз на роль `aml_app`;
- конкретные пользователи/логины получают эту роль;
- локальные пароли не нужно хранить в основном SQL-файле.

### Helper-функции для RLS

В security-слое есть функции:

- `fn_get_current_user_id()`;
- `fn_is_project_member(project_id)`;
- `fn_is_project_admin(project_id)`;
- `fn_can_access_issue(issue_id)`.

`fn_get_current_user_id()` читает значение из настройки сессии:

```sql
app.current_user_id
```

Приложение перед запросами должно установить текущего пользователя:

```sql
SELECT set_config('app.current_user_id', '<user-uuid>', false);
```

После этого RLS-политики понимают, от имени какого пользователя выполняется запрос.

### Представления

Создаются views:

- `v_users_public`;
- `v_active_projects`;
- `v_project_issues`;
- `v_my_notifications`;
- `v_active_sessions`.

Зачем они нужны:

- скрыть чувствительные поля;
- убрать удаленные записи;
- дать приложению удобные read-модели;
- не заставлять фронт/бэкенд каждый раз собирать одно и то же JOIN-выражение.

### RLS-Политики

Row Level Security включен почти на всех бизнес-таблицах.

Примеры правил:

- пользователь видит только проекты, где он участник;
- пользователь видит задачи только доступных ему проектов;
- пользователь может обновлять только свой профиль;
- пользователь видит только свои уведомления;
- пользователь может писать time log только за себя;
- администратор имеет полный доступ;
- приложение не имеет прямого доступа к `audit_log`.

Это важная часть модели: безопасность не полностью зависит от приложения. Даже если приложение ошибется в WHERE-условии, база дополнительно ограничит строки через RLS.

## Аудит, активность и версионирование

### `activity_log` И `audit_log` Решают Разные Задачи

`activity_log` - бизнес-события. Он отвечает на вопрос:

```text
Что произошло с точки зрения пользователя?
```

Например:

- создан проект;
- создана задача;
- изменен статус.

`audit_log` - техническая история изменений. Он отвечает на вопрос:

```text
Какая строка в какой таблице была изменена и как именно?
```

Оба журнала нужны, потому что один не заменяет другой.

### Как работает audit

В `03_functions_and_triggers.sql` создается функция `fn_audit_trigger()`.

Она срабатывает после:

- `INSERT`;
- `UPDATE`;
- `DELETE`.

Для каждой операции пишется строка в `audit_log`.

Для определения пользователя используется:

```sql
aml_task.fn_get_current_user_id()
```

Поэтому перед изменениями полезно выставлять:

```sql
SELECT set_config('app.current_user_id', '<user-uuid>', false);
```

### Почему audit не хранит парольные хэши и refresh token hash

Если аудит сохранял бы все поля без исключений, чувствительные значения дублировались бы в другой таблице.

Поэтому из audit JSON удаляются:

- `password_hash`;
- `refresh_token_hash`.

Это снижает риск утечек и соответствует принципу минимизации чувствительных данных.

## Короткое резюме

Эта база описывает таск-трекер с проектами, задачами, спринтами, ролями, аудитом и защитой доступа на уровне PostgreSQL.

Ключевые инженерные идеи:

- проект является границей доступа;
- задачи нумеруются внутри проекта;
- важные сущности удаляются мягко через `deleted_at`;
- статусы нормализованы и имеют категории;
- история статусов хранится отдельно от текущего статуса;
- audit log отделен от business activity log;
- секреты не коммитятся;
- RLS защищает данные даже при ошибках в приложении;
- база активно проверяет бизнес-инварианты через constraints.
